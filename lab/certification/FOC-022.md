# FOC-022 — تحويل فائض العجز إلى الغداء (Overflow)

> **الحالة: CERTIFIED** · تاريخ الشهادة: 2026-07-22 · مُولَّد من تشغيلٍ حقيقيّ في المختبر الدستوري.

## القصّة
يُقبَض تحصيلُ ذمّةٍ موجَّهٌ للعجز بمبلغ 9000 (المتبقّي 3000). يُسدَّد العجز المشترك بالكامل (يصير 0)، ويتحوّل الفائض 6000 تلقائيًّا إلى خزينة الغداء.

## الشرح التجاري (بالعربية المبسّطة)
المبلغ الموجَّه للعجز أكبر من العجز المتبقّي: أغلق خزينة العجز المشتركة كاملًا (−3000 → 0)، والفائض 6000 تحوّل تلقائيًّا إلى الغداء بقاعدة قراءةٍ لا صفوف تُكتب (القانون 2 — الاشتقاق، القانون 9 — حدّ العجز). فصارت خزينة الغداء 0 → 6000، وبقي الديوان دون تغيّر.

## المقارنة (قبل ↔ بعد)

| القيمة | قبل | بعد |
|---|---|---|
| خزينة الغداء | 0 | 6000 |
| خزينة الديوان | 5000 | 5000 |
| العجز المتبقّي | -3000 | 0 |
| رصيد LAB-004 | 0 | -9000 |
| سجلّ التبرّعات (عدد) | 0 | 0 |
| سجلّ القيود (audit) | 0 | 0 |

## المتوقَّع ↔ الفعلي (كل تحقّق)

- ✅ العجز المتبقّي −3000 → 0 (سُدّد بالكامل) — `-3000 → 0`
- ✅ الفائض المتحوّل للغداء = 6000 — `overflow=6000`
- ✅ خزينة الغداء 0 → 6000 (استلمت الفائض) — `0 → 6000`
- ✅ الديوان لم يتغيّر — `5000`

## ملاحظة — قرار مطلوب من المالك (POSSIBLE BUG / OWNER DECISION)
لتشغيل قاعدة الفائض لزم توجيهُ مبلغٍ للعجز أكبر من العجز المتبقّي (9000 > 3000). لا يملك أيُّ عضوٍ ذمّةً تاريخيةً تفوق 3000، فاختير عضوٌ مسدَّد (LAB-004، ذمّته التاريخية = 0) موجَّهًا للعجز؛ فصُنّف المبلغ «تحصيل ذمّة تاريخية» (ق4). قاعدة تحويل الفائض — موضوع هذا الفصل — عملت عملًا صحيحًا وموثَّقًا. لكنّ أثرًا جانبيًّا ظهر: تحصيلُ الذمّة (ق4) لا يُحَدُّ برصيد العضو التاريخيّ الفعليّ، فانخفض رصيد LAB-004 إلى −9000 (رصيدٌ دائنٌ كبير لا يقابله دَينٌ حقيقيّ). هذا سيناريو مُختبَريّ مُصطنَعٌ لإظهار الفائض، لا عمليةٌ طبيعية. **قرار مطلوب من المالك:** هل يُحَدُّ تحصيلُ الذمّة التاريخية عند رصيد العضو الفعليّ (منعُ رصيدٍ دائنٍ زائف)؟ لا يُصلَح شيء في الإنتاج قبل قرارك — رُصِدَ فقط دون تعديل.
