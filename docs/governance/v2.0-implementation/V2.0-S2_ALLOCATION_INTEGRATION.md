# V2.0 · Slice 2 — Allocation Integration (BEHIND FLAG · flag OFF)

**Phase:** MODEL2 V2.0 Implementation · **Slice:** 2 · **Date:** 2026‑07‑23 · **Baseline:** `main` @ `2d4f300` (after Slice 1 PR #159 merged)

## Objective
Connect the Slice‑1 pure allocation engine (`public/js/allocation-engine.js`) to the two live paths where
obligations are created, implementing the frozen **OD‑01** and **OD‑02** behaviour — while remaining
byte‑identical to today until the flag is enabled in Slice 7.
- **OD‑01** (member payment) — every member subscription payment records its ordered allocation across the
  member's outstanding obligations (CA‑001 order), producing a permanent, immutable allocation record of
  the same audit quality as a payment.
- **OD‑02** (credit consumption) — at the creation of a new legal obligation (annual dues generation), an
  existing member credit is consumed against obligations in the **same** constitutional order, generating an
  equal‑quality allocation record. Consumption happens **only** at obligation creation, never standalone.

## Behaviour introduced (only when flag ON — Slice 7)
- A member subscription payment computes its allocation (current‑year → historical → future → credit) and
  inserts immutable rows into `allocation_records`.
- Annual‑dues generation consumes any pre‑existing member credit against the freshly created obligations in
  the same order, inserting rows tagged `source_kind: 'credit_consumption'`.
- Recording is **best‑effort metadata**: it never alters balances (the certified net remains the single
  source of truth) and never blocks or fails the core operation (wrapped in try/catch, guarded on the
  engine's presence and a positive amount). Forward‑only — no historical voucher is ever reallocated.

## Migration impact
None. Forward‑only, additive. No Supabase schema change is applied in this slice and no historical data is
transformed. The `allocation_records` writes occur **only** when the flag is enabled (Slice 7); with the flag
OFF every recorder is an immediate no‑op, so the DB is untouched and net balances are unchanged. The Slice‑1
migration snapshot (`main` @ `c7c2aaa`) remains the reconciliation reference for activation.
