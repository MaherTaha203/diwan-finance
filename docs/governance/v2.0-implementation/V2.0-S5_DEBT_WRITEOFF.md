**Phase:** MODEL2 V2.0 Implementation · **Slice:** 5 · **Date:** 2026‑07‑23 · **Baseline:** `main` @ `7a7f3c0`
**Constitutional basis:** CA‑007 (rule 2) · V2.0‑DIS OD‑07.

## 1 · Objective
Implement the frozen **Debt Write‑off** for a member's **permanent departure or death**: resolve the member's
**outstanding receivable to zero** via an **explicit, audited, NON‑CASH** member‑ledger transaction — never a
silent deletion of balances (Law 1/5/6). Delivered pure + flag‑guarded so the system is **byte‑identical to
today** until Slice 7 activation. (Credit Write‑off — CA‑007 rule 3 — is Slice 6.)

## 3 · Behaviour introduced (only when flag ON — Slice 7)
`BusinessOps.writeOffDebt({memberId, reason})` records a NEW `debt_write_off` (non‑cash) for a departed
member's full outstanding debt, driving `finalBalance` to zero. It moves **no** cash treasury (Law 8
untouched), preserves all prior transactions (Law 5), and is fully audited (Law 6). Deactivation
(BO‑09/FOC‑017) remains the separate administrative register change; the write‑off is the **explicit money
event** that accompanies permanent closure.

## 4 · Migration impact
None. Forward‑only, additive; the write‑off reuses the existing `receipts` voucher table with a dedicated
non‑cash `movement_type` (the `donation_inkind` non‑cash precedent) and a neutral `fund_type:'writeoff'` so it
never appears in a cash/treasury/donation view. No schema change. Flag OFF → `E_DISABLED` and no live UI path
calls BO‑12, so the DB is untouched and behaviour is byte‑identical.
