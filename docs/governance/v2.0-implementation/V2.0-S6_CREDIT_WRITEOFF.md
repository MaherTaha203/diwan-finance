**Phase:** MODEL2 V2.0 Implementation · **Slice:** 6 · **Date:** 2026‑07‑23 · **Baseline:** `main` @ `7a7f3c0` (stacked on Slice 5, PR #163 not yet merged)
**Constitutional basis:** CA‑007 (rule 3/4) · OD‑06 · OD‑07.

## 1 · Objective
Implement the frozen **Credit Write‑off** for a member's **permanent departure or death**: resolve the
member's **outstanding credit to zero** via an **explicit, audited, NON‑CASH** member‑ledger transaction.
Member credit is **never auto‑refunded** and must **not remain a perpetual liability** (OD‑06/OD‑07). Pure +
flag‑guarded → **byte‑identical to today** until Slice 7. This completes the CA‑007 write‑off pair (Debt =
Slice 5, Credit = Slice 6).

## 3 · Behaviour introduced (only when flag ON — Slice 7)
`BusinessOps.writeOffCredit({memberId, reason})` records a NEW `credit_write_off` (non‑cash) for a departed
member's full outstanding credit, driving `finalBalance`/`creditBalance` to zero. It moves **no** cash
treasury (Law 8 untouched), performs **no refund** (OD‑06/OD‑07), preserves all prior transactions (Law 5),
and is fully audited (Law 6). Together with Slice 5's Debt Write‑off, a departed member's account is closed
only through **explicit financial transactions** — never a silent balance edit (CA‑007 rule 5).

## 4 · Migration impact
None. Forward‑only, additive; reuses the `receipts` voucher table with a dedicated non‑cash `movement_type`
and neutral `fund_type:'writeoff'` (never shown in a cash/treasury/donation view). No schema change. Flag OFF
→ `E_DISABLED`, no live UI path calls BO‑13, and the fin.js term is inert with zero rows.
