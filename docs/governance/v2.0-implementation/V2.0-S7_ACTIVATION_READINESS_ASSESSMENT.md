## 2 · The blockers (verified against the live schema)

### B1 · Allocation recording target table is missing (Slice 2)
`allocation-integration.js` writes to **`SB.from('allocation_records')`** (2 call sites). **No such table
exists.** The nearest table is Phase‑15 **`payment_allocations`** (0 rows), but:
- **Different columns:** `payment_allocations(receipt_id, member_id, target_type∈{subscription,historical,credit}, target_year, amount_ils, rebuild_run_id)` vs the Slice‑1 `buildAllocationRows` output (`obligation_id, obligation_kind, year, amount_allocated, source_ref, …`).
- **Different (conflicting) order:** its comment declares *"Order: oldest sub → next sub → historical →
  credit"* — which **contradicts the constitutionally‑amended CA‑001 order** (Current‑Year → Historical →
  Future → Credit). → a **constitutional reconciliation** decision, not just a schema one.
- Effect if activated as‑is: the insert throws; Slice 2's recorders are best‑effort `try/catch`, so allocation
  recording would **silently no‑op** — "active" but recording nothing.

### B2 · Refund voucher cannot be written (Slice 4 · BO‑11)
`refundReceipt` inserts into **`payments`** a row with `movement_type:'refund'`, `origin_receipt_id`,
`linked_receipt`, `is_refund`, `fund_type:(origin)`, but **no** `beneficiary_type` / `expense_type`. The live
`payments` table:
- has **no** `origin_receipt_id`, `linked_receipt`, `is_refund` columns;
- **`beneficiary_type` NOT NULL** (CHECK `member|manual`) — not set by the refund payload → violation;
- **`expense_type` NOT NULL** (no default) — not set → violation;
- **`fund_type` CHECK `food|diwan`** — a refund of a `donation` receipt cannot be stored.
- Effect if activated: **every refund fails** with `E_WRITE` (admin‑visible error).

### B3 · Write‑off vouchers cannot be written (Slices 5/6 · BO‑12/13)
`writeOffDebt` / `writeOffCredit` insert into **`receipts`** with `movement_type:'debt_write_off'|'credit_write_off'`,
`fund_type:'writeoff'`, `is_write_off`, `destination_treasury:null`, and **no** `payer_type`. The live
`receipts` table:
- **`fund_type` CHECK `food|diwan|donation`** — `'writeoff'` is rejected → violation;
- **`payer_type` NOT NULL** (CHECK `member|contact|manual`) — not set → violation;
- has **no** `is_write_off` column.
- Effect if activated: **every write‑off fails** with `E_WRITE`.

## 3 · Why this is a STOP (Owner decisions required)
1. **Production schema migration** on live financial tables (63 receipts · 28 payments · 151 members) — a
   **business/infrastructure decision**. The V2.0 roadmap lists "migration verification" as a Slice‑7 gate;
   this is that migration, and it must be Owner‑approved before it touches production.
2. **Constitutional reconciliation (B1 order):** the existing `payment_allocations` order contradicts the
   amended CA‑001 order. Whether MODEL2 V2.0 *creates a new* `allocation_records` table or *re‑targets and
   corrects* `payment_allocations` (and what happens to its Phase‑15 rebuild machinery) is a **constitutional +
   architectural** decision.

## 4 · Options for the Owner (no work done — for decision only)
**For B1 (allocation storage):**
- **A1 — New `allocation_records` table** matching `buildAllocationRows` (forward‑only, additive; leaves
  Phase‑15 `payment_allocations` untouched). Cleanest; two allocation stores coexist (new = MODEL2 audit).
- **A2 — Re‑target + correct `payment_allocations`** to the CA‑001 order and adapt `buildAllocationRows` to
  its columns. Reuses one store but **changes** certified Phase‑15 semantics (heavier; re‑cert needed).

**For B2/B3 (refund + write‑off storage):**
- **S1 — Extend existing tables:** add the missing columns to `payments`/`receipts`, relax the `fund_type`
  CHECKs (or map refunds/write‑offs onto allowed values), and make `beneficiary_type`/`expense_type`/`payer_type`
  nullable or defaulted for these movement types.
- **S2 — Dedicated tables** (`refunds`, `member_write_offs`) so the core voucher tables and their constraints
  stay untouched; the read models (`fin.js`/`fin2.js`) read the dedicated tables. Cleaner isolation; more code.

## 5 · Recommendation (for consideration — not a decision)
- **B1 → A1** (new `allocation_records` table): forward‑only and additive, keeps the frozen CA‑001 order
  authoritative, and does not disturb the certified Phase‑15 `payment_allocations`/rebuild path.
- **B2/B3 → S2** (dedicated `refunds` + `member_write_offs` tables): avoids relaxing the money‑critical
  CHECK/NOT‑NULL constraints on the certified `receipts`/`payments` tables, and keeps refund/write‑off records
  cleanly separated for audit. (This means small code edits to point BO‑11/12/13 and the read models at the
  dedicated tables — a follow‑up slice, still flag‑gated.)
