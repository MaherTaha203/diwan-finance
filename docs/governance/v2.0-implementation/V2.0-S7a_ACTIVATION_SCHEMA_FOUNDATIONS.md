# V2.0 · Slice 7a — Activation Schema Foundations (owner‑approved migration · flag OFF)

**Phase:** MODEL2 V2.0 Implementation · **Slice:** 7a (pre‑activation) · **Date:** 2026‑07‑23 · **Baseline:** `main` @ `a015e8d`
**Owner decision:** B1 → **A1** (dedicated `allocation_records`) · B2/B3 → **S2** (dedicated `refunds` + `member_write_offs`). Do **not** repurpose `payment_allocations`; do **not** extend `payments`/`receipts`.

## 1 · Objective
Resolve the Slice‑7 activation blockers by adding the owner‑approved **dedicated tables** to the live Supabase
schema and pointing the flag‑gated code at them — **without flipping the flag**. Activation remains blocked
(a separate PR). The tables are empty, so with the flag OFF the system is **byte‑identical to today**.

## 2 · Migration applied (Supabase `ralifvemgapmsgrjgazh`)
Migration `model2_v2_activation_tables` — additive only; **no existing table altered**. Three tables, each
**RLS‑enabled** mirroring the certified `receipts`/`payments` posture (`is_admin()` writes,
`is_provisioned_user()` reads, `anon` blocked):
- **`allocation_records`** (B1→A1) — MODEL2 ordered‑allocation audit (CA‑001 order). Columns match Slice‑1
  `buildAllocationRows` (`source_ref, source_kind, member_id, obligation_id, obligation_kind, year,
  amount_allocated, allocated_at, immutable`). Insert allowed to any provisioned user (written on the normal
  payment‑save path); immutable audit — balances still derive from `receipts`, never this table.
- **`refunds`** (B2→S2) — first‑class refund movement (CA‑005/BO‑11); linked to the origin receipt, funded
  from the origin treasury. `payments` untouched.
- **`member_write_offs`** (B3→S2) — debt/credit write‑offs at member closure (CA‑007/BO‑12/13), non‑cash.
  `receipts` untouched.

**Persistence + retrieval verified on the live schema:** a probe row matching each BO's exact insert payload
was inserted, read back, and deleted; all three tables end at **0 rows**.

## 3 · Code adapted to the dedicated tables
| File | Change |
|---|---|
| `public/js/operations.js` | **BO‑11** now inserts into `refunds` (no `RFND‑…`), reads prior refunds from `DB.refunds`. **BO‑12/13** now insert into `member_write_offs` (no `WO‑`/`CWO‑`), number from `DB.member_write_offs`. `logAction` table names updated. `payments`/`receipts` no longer touched by these operations. |
| `public/js/fin.js` | `memberStatement` reads debt/credit write‑offs from **`DB.member_write_offs`** (was `DB.receipts`). |
| `public/js/fin2.js` | `treasuryBalance` subtracts each live refund (from **`DB.refunds`**) from its origin treasury — the outflow is no longer visible via `payments`, so it is applied here (Law 8). |
| `public/js/data.js` | `loadAllData` now also loads `refunds` + `member_write_offs` into `DB` (read models depend on them). `allocation_records` is write‑only (nothing reads it) — not loaded. |
| `public/js/allocation-integration.js` | Unchanged — already writes `allocation_records`; the table now exists and matches the row shape. |

## 4 · Behaviour / migration impact
**Zero behaviour change.** The flag stays **OFF**, so BO‑11/12/13 still return `E_DISABLED` and no live path
writes; `allocation-integration` recorders are no‑ops. The new tables are **empty**, so every read model
(`fin.js` write‑offs, `fin2.js` refund outflow) contributes nothing. No existing table or row was modified.
