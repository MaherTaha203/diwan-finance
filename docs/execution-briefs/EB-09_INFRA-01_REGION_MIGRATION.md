# Execution Brief — EB-09 · INFRA-01 — تقريب المسافة: تهجير قاعدة البيانات إلى إقليمٍ أوروبيّ
# Production Evolution · Performance (Infrastructure) — Supabase region migration (Tokyo → Frankfurt)

> **الحالة:** خطةٌ مقترحة — **لا تنفيذَ قبل اعتماد المالك** (تمسّ بنية الإنتاج: مصادقة + بيانات).
> **الدافع:** بعد مرحلتَي تسريع الدخول (PR #52، #53) بلغنا سقف المتصفح؛ الرحلة الواحدة إلى طوكيو ≈ 350–450ms.
> نقل المشروع إلى **فرانكفورت (eu-central-1)** يجعلها ≈ **60–90ms** — تحسّنٌ ~4–5× في كل نداءٍ للنظام (دخول، حفظ، تحديث).

## 0) حقيقة تقنية حاكمة
إقليم مشروع Supabase **لا يتغيّر في مكانه**. الطريق الوحيد المدعوم: **مشروعٌ جديدٌ في الإقليم الأقرب + تهجيرٌ كامل + تبديل عنوانٍ في الكود**. المشروع القديم يبقى سليماً كما هو حتى بعد النجاح (خطّ رجوعٍ فوريّ).

## 3) الضمانات
- **القديم لا يُمسّ ولا يُحذف** — كل الخطوات 2–7 بناءٌ موازٍ؛ الخطوة 8 وحدها «تُقلب» وهي **commit واحدٌ قابلٌ للعكس فوراً**.
- **byte-parity ماليّ إلزاميّ** قبل القطع (نفس منهج Foundation-B).
- كلمات المرور تبقى — لا إزعاج للمستخدمين الأربعة.
- الفجوة الوحيدة المحتملة: سندٌ يُدخل أثناء النافذة على القديم بعد النسخ — لذلك التجميد القصير (أو نسخ فرقٍ نهائيّ قبل القطع مباشرة).

## 4) قرارات مطلوبة من المالك قبل التنفيذ
1. **اعتماد الإقليم**: فرانكفورت `eu-central-1` (توصيتي — الأقرب المتاح والأكثر استقراراً لبلاد الشام). بديل: لندن/زيورخ.
2. **نافذة التجميد** (~30 دقيقة، أي وقتٍ يناسبك).
3. مَن ينفّذ خطوة المرفقات/SMTP اليدوية (أنت عبر اللوحة بإرشادي، أو تزوّدني بصلاحيةٍ مؤقتة).

**عند اعتمادك:** أنفّذ 2→7 فوراً (ساعة تقريباً، بلا أي أثرٍ على الإنتاج)، أسلّمك تقرير المطابقة، ثم ننفّذ القطع (8) في نافذتك.
